## Tech Stack Details

### Frontend
**React 19 + Vite + TypeScript**
- Why: Industry standard. Used by Meta, Vercel, Stripe
- Vite: 10x faster than CRA, modern tooling
- Trade-off: None. This is the right choice

**3D Graphics: Three.js**
- Battle-tested. Used by Sketchfab, Google (Earth), Mozilla
- Alternative: Babylon.js (also good, picked Three.js for community size)

**State Management: TanStack Query + Zustand**
- TanStack Query: Server state management (industry moving this way)
- Zustand: Light client state (not Redux complexity)
- Why: Reduces bugs, simpler than Redux, what modern teams use (Airbnb, Discord)

**Styling: Tailwind CSS**
- Utility-first CSS (Vercel, GitHub, Stripe use this)
- Better maintainability than traditional CSS/SCSS

**Testing: Vitest + React Testing Library**
- Vitest: Faster than Jest (same syntax)
- React Testing Library: Tests user behavior, not implementation

---

### Backend
**FastAPI (Python 3.11+)**
- Why:
  - Fastest Python framework (2x speed of Django)
  - Automatic OpenAPI docs (Swagger)
  - Built-in async/await (can handle 1000+ concurrent connections)
  - Used by Netflix internal tools, Uber Eats backend components
- Alternative considered: Node.js Express—FastAPI is faster for this workload

**Database: PostgreSQL 15+**
- ACID compliance for satellite data consistency
- JSON support for flexible schemas (not enforced)
- Proven at any scale (Stripe, Slack, GitHub use PostgreSQL)
- Alternative: MySQL—but PostgreSQL is better for this use case (JSON, full-text search, arrays)

**ORM: SQLAlchemy 2.0**
- Industry standard for Python
- Mature, excellent async support
- Migrations: Alembic (battle-tested)

**Authentication: JWT + OAuth2 (built into FastAPI)**
- Why: Stateless, scalable, industry standard
- No third-party needed (free)
- Supports future token rotation, multi-device login

**Task Queue: Celery + Redis**
- Satellite pass predictions run in background (long-running)
- Industry standard (Spotify, Pinterest, Instagram use Celery)
- Redis: Free, single-instance setup for MVP

### Infrastructure & DevOps

**Container: Docker**
- Multi-stage builds for optimization
- Non-root user (security)
- ~150MB final image size target

**Orchestration: Docker Compose (MVP) → Kubernetes (when needed)**
- Compose: Easy local dev and small deployments
- Kubernetes: When you need auto-scaling (future, not day 1)

**CI/CD: GitHub Actions**
- FREE for public repos ✓
- FREE for private repos: 2,000 minutes/month (enough for this project)
- What it will do:
  - Run tests on every PR
  - Run linter/formatter checks
  - Build and test Docker images
  - Deploy to staging on `main` branch
  - Security scanning (Dependabot—free)

**Monitoring: Prometheus + Grafana**
- Both free, open-source, industry standard
- Prometheus scrapes FastAPI metrics
- Grafana dashboards for visualization
- No cost at any scale

**Logging: JSON structured logs → stdout (Docker collects)**
- Approach: Docker logs everything, structured as JSON
- Option 1 (Free): Parse logs with open-source ELK (Elasticsearch, Logstash, Kibana)
- Option 2 (Better for MVP): Use free tier of Grafana Loki (~3GB/day free)
- Why JSON: Easy to search, filter, aggregate

**Secrets Management: GitHub Secrets (free)**
- Store API keys, DB credentials in GitHub Secrets
- Accessed by Actions, not committed to repo

**Container Registry: GitHub Container Registry (free)**
- Store Docker images alongside code
- 5GB free storage (plenty for this)
- Alternative: Docker Hub (also free, 1 private repo)

## Phase 1: Production-Ready (8-12 weeks)

### Week 1-2: CI/CD Pipeline
```yaml
GitHub Actions Workflows:
- test.yml: Run pytest on backend, npm test on frontend
- lint.yml: Black + pylint (backend), ESLint (frontend)
- docker.yml: Build and push Docker images
- security.yml: Dependabot alerts (GitHub native)
```
**Effort**: 20 hours. **Benefit**: Catch bugs before they hit main.

### Week 3-4: Monitoring Foundation
```
Backend metrics:
- FastAPI middleware exports: request_count, request_duration, error_rate
- Database: connection pool size, query duration
- Business: satellite passes predicted, alerts triggered

Prometheus config: Scrape FastAPI every 15 seconds
Grafana dashboards: System health, error rates, API latency

Local first: Docker Compose includes Prometheus + Grafana
```
**Effort**: 15 hours. **Benefit**: See what's breaking in real-time.

### Week 5-6: Structured Logging
```python
# Before:
logger.info("Satellite pass predicted")

# After (JSON structured):
logger.info("satellite_pass_predicted", extra={
    "satellite_id": "ISS",
    "pass_confidence": 0.95,
    "duration_seconds": 420,
    "user_id": "abc123"
})
```
**Effort**: 10 hours. **Benefit**: Search/filter logs by any field.

### Week 7-8: API Hardening
- JWT auth with expiration (15 min access, 7 day refresh tokens)
- Rate limiting: FastAPI Slowapi (100 req/min per IP)
- Error standardization: All errors return `{ "error": "code", "message": "...", "request_id": "..." }`
- API versioning: `/api/v1/passes` (future v2 safe to deploy)

**Effort**: 12 hours. **Benefit**: Production-grade security.

### Week 9-10: Database Resilience
- Connection pooling: SQLAlchemy pool_size=20, max_overflow=40
- Read replicas ready (config in place, not deployed yet)
- Backup automation: pg_dump daily → S3 (free tier)
- Migration testing in CI: Every migration runs test DB first

**Effort**: 8 hours. **Benefit**: Data safety, high availability ready.

### Week 11-12: Documentation & Runbooks
- Deployment guide (1-command deploy)
- Incident response guide (what to do if API is down)
- Monitoring alert thresholds (what triggers page/Slack)
- Rollback procedures

**Effort**: 10 hours. **Benefit**: Anyone on team can deploy safely.

## Architecture Decisions (Why Not X?)

| Option | Why Not | Our Choice |
|--------|---------|-----------|
| Node.js backend | Slower than FastAPI for CPU-heavy orbital math | **FastAPI** |
| MongoDB | No transactions, doesn't fit relational satellite data | **PostgreSQL** |
| Redux state | Overkill, TanStack Query handles 80% of needs | **Zustand + TanStack Query** |
| Jenkins CI/CD | Needs separate server, GitHub Actions simpler | **GitHub Actions** |
| ELK Stack | Too heavy for MVP (3 services to manage) | **Prometheus + Grafana** |
| Heroku | Now expensive; we use Docker + free cloud tier | **Docker + GitHub Actions** |
| REST only | Satellite tracking needs real-time updates | **REST + WebSocket** |

## Deployment Architecture (MVP → Scale)

**MVP (Day 1)**
```
1 FastAPI instance + 1 PostgreSQL + 1 Redis
Docker Compose on single $5-10/month VPS (DigitalOcean, Linode, Hetzner)
Backup: Daily PostgreSQL dumps to S3
```

**Growth (100k users)**
```
3x FastAPI replicas (behind nginx load balancer)
PostgreSQL primary + read replica
Redis cluster (2 replicas)
CDN for frontend (Cloudflare—free tier)
```

**Scale (1M+ users)**
```
Kubernetes (EKS on AWS, GKE on Google Cloud)
PostgreSQL managed (RDS—handles backups)
Redis managed (ElastiCache)
S3 for static assets + backups
```

**Key principle**: Everything we do now scales to production. No rearchitecting later.

## Mentality Shift: Senior Engineer Perspective

**What separates "works" from "production":**

1. **Observability > Features** — If you can't see it breaking, you can't fix it
2. **Gradual rollout** — 5% of traffic to new code first, not 100%
3. **Backward compatibility** — Never remove an API endpoint, deprecate it
4. **Automate everything** — If you do it twice, script it
5. **Boring is good** — Use proven tech, not shiny new frameworks
6. **Test the happy path and sad path** — Most bugs hide in error cases
7. **Document the "why"** — Code explains "what", docs explain "why we chose this"
